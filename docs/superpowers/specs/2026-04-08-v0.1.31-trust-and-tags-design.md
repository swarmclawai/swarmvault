# v0.1.31 "Trust & Tags" Design Spec

## Summary

Three features that make the vault more trustworthy and navigable:
1. **Contradiction detection** ‚Äî surface conflicting claims across sources
2. **Semantic auto-tagging** ‚Äî broad domain tags extracted alongside concepts
3. **Approval diffs** ‚Äî section-level summaries and unified diffs for review bundles

## Feature 1: Contradiction Detection

### Deterministic Pass (compile time, always runs)

After all source analyses are collected in `compileVault()`, run a cross-source claim comparison:

1. Collect all `SourceClaim` objects across all analyses
2. For each pair of claims from different sources, compute topic similarity (fuzzy text match on claim text) and check for opposite polarity (`positive` vs `negative`)
3. When a conflict is found:
   - Mark both claims' `status` as `"conflicted"` in the analysis
   - Add a `conflicted` graph edge between the two source nodes with `evidenceClass: "ambiguous"`, both claim texts in provenance, and confidence derived from the delta between the two claims' confidence scores
4. Surface conflicts in `wiki/graph/report.md` under a new "Contradictions" section listing each conflict with source references, claim texts, and confidence delta

**Implementation location**: New function `detectContradictions()` in `vault.ts`, called after `buildGraph()` but before `buildGraphOrientationPages()`. Modifies the graph in place by adding `conflicted` edges and updating claim statuses.

**Topic similarity**: Use normalized lowercase token overlap (Jaccard similarity) with a threshold of 0.3. No embedding or LLM call required. This catches "The system uses AES-256 encryption" vs "The system does not use encryption" but won't catch semantically distant contradictions ‚Äî those are for deep lint.

### LLM-Powered Pass (lint --deep)

Extend the existing deep lint prompt in `deep-lint.ts`:

1. The deep lint prompt already has access to compiled pages and the graph report
2. Add explicit instruction: "Identify claims that contradict each other across different sources. For each contradiction, include both source IDs, the conflicting claim texts, and your confidence that this is a genuine contradiction vs a nuance difference."
3. Upgrade the existing `"contradiction_candidate"` lint code to `"contradiction"` and require `relatedSourceIds` to include both conflicting sources
4. Keep `"contradiction_candidate"` as a separate code for cases where the LLM suspects a contradiction but isn't confident

### CLI Surface

- `swarmvault lint --conflicts`: shortcut that runs structural lint filtered to contradiction/conflicted findings only. No `--deep` required for deterministic contradictions. If `--deep` is also passed, includes LLM-detected contradictions too.
- `--json` output includes contradiction findings in the standard `LintFinding[]` format

### Graph Workspace

- Conflicted edges get a distinct visual treatment (red/dashed) in `graph serve`
- The graph report panel shows the Contradictions section

## Feature 2: Semantic Auto-Tagging

### Analysis Schema Change

In `packages/engine/src/analysis.ts`:

1. Add `tags` field to `sourceAnalysisSchema`:
   ```
   tags: z.array(z.string()).max(5).default([])
   ```
2. Add instruction to the analysis system prompt:
   "Return up to 5 broad domain tags that categorize this source. Tags should be lowercase kebab-case (e.g., cryptography, distributed-systems, machine-learning). These are broader categories, not specific concepts or entity names."

### Type Change

In `packages/engine/src/types.ts`:

1. Add `tags: string[]` to `SourceAnalysis` interface

### Tag Propagation

In `packages/engine/src/vault.ts` and `markdown.ts`:

1. During compile, pass analysis tags into `GeneratedPageDecorations.extraTags` so `decoratedTags()` includes them in page frontmatter
2. In `buildGraph()`, propagate tags to `GraphNode` metadata so graph query/explain can surface them
3. Tags flow into search index automatically since pages index all frontmatter
4. The `similarityReasons` on graph edges already supports `"shared_tag"` ‚Äî semantic tags will improve similarity edge quality

### Heuristic Provider

The heuristic provider returns empty tags array ‚Äî no regression for offline use. Tags only populate when a real LLM provider is configured.

### Files to modify

| File | Change |
|------|--------|
| `packages/engine/src/analysis.ts` | Add `tags` to schema and prompt |
| `packages/engine/src/types.ts` | Add `tags: string[]` to `SourceAnalysis` |
| `packages/engine/src/vault.ts` | Pass tags through to page decorations and graph nodes |
| `packages/engine/src/markdown.ts` | Tags flow through existing `decoratedTags()` ‚Äî minimal change |

## Feature 3: Approval Diffs

### Summary View (default `review show`)

In `packages/engine/src/vault.ts` (or a new `review.ts` if extraction is cleaner):

1. For each `ApprovalEntryDetail`, compare `currentContent` vs `stagedContent`
2. Parse frontmatter and body sections from both versions
3. Produce a section-level summary per entry:
   - "Added 3 concepts, updated summary, changed 2 claims, removed 1 entity link"
   - For `changeType: "create"`: "New page: {title} ({kind})"
   - For `changeType: "promote"`: "Promoted from candidate: {title}"
   - For `changeType: "delete"`: "Removed: {title}"
4. Return summaries as a new `changeSummary: string` field on `ApprovalEntryDetail`

### Full Diff View (`review show --diff`)

1. Compute unified diff between `currentContent` and `stagedContent` with 3 lines of context
2. Implement a simple line-by-line unified diff (longest common subsequence algorithm). No external dependency ‚Äî the diff is straightforward for markdown pages.
3. Output standard `---/+++` format to terminal
4. For `--json`, include `diff: string` field alongside the summary

### Per-Entry Accept/Reject

1. Add `--entry <pageId>` flag to `review accept` and `review reject` commands
2. When `--entry` is passed, update only the matching entry's `status` in the manifest (not the whole bundle)
3. Write the updated manifest back to `state/approvals/{id}/manifest.json`
4. When all entries in a bundle are resolved (all accepted or all rejected), auto-resolve the bundle
5. Mixed resolution (some accepted, some rejected) applies only the accepted entries

### Graph Workspace

The viewer approval panel shows the section-level `changeSummary` inline. Full diff remains CLI-only.

| File | Change |

| `packages/cli/src/index.ts` | Add `--diff` flag to `review show`, `--entry` flag to `review accept`/`reject` |

## Testing

### Engine Tests

1. **Contradiction detection**: Create two analyses with opposing polarity claims on the same topic. Verify `detectContradictions()` produces conflicted edges and updates claim statuses.
2. **No false positives**: Create two analyses with same-polarity claims on the same topic. Verify no conflicted edges.
3. **Semantic tags**: Mock a provider that returns tags in structured output. Verify tags appear in page frontmatter and graph nodes.
4. **Heuristic tags**: Verify heuristic provider returns empty tags without error.
5. **Approval diff summary**: Create an approval bundle with known current/staged content. Verify `computeChangeSummary()` produces accurate section-level descriptions.
6. **Approval diff unified**: Verify unified diff output matches expected `---/+++` format.
7. **Per-entry accept**: Accept one entry in a multi-entry bundle. Verify only that entry's status changes. Verify bundle auto-resolves when all entries are resolved.

### Smoke Validation

- Run the heuristic smoke lane to verify no regression from the new analysis schema field
- Run a provider-backed smoke with a multi-source vault to verify tags populate and contradictions surface
